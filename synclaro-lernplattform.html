Wie denkt eine KI? — Next-Word-Prediction

🔤 Das Grundprinzip: Drei Wörter

Janine's Erklärung (vom Coach als perfekt bewertet):

"KI ist eine riesige Datenbank, die Informationen miteinander in Verbindung setzt und versucht herauszufinden, was die wahrscheinlichste Fortsetzung ist."

KI ist keine Magie — sie ist Wahrscheinlichkeitsrechnung auf einer Wort-Korrelationstabelle. Jedes Wort kostet einen Transformer-Durchlauf.

Wort-PaarVorkommen
„Himmel" + „Blau"318 ×
„Finanzamt" + „glücklich"7 ×
→ KI wählt immer die höchste Zahl—

Das Modell wurde mit allen Texten der Welt trainiert — Internet, Bücher, Twitter, Gerichtsurteile.

💡 Kernlearning
  • KI folgt immer der höchsten Wahrscheinlichkeit — nicht der Wahrheit
  • Das erklärt seltsame Fehler: die KI „halluziniert" wahrscheinliche, aber falsche Antworten
  • Nicht deterministisch: gleiche Frage → leicht unterschiedliche Antworten (Temperatur)

Thinking & Reasoning Modelle

⚡ Standard-Modus

Prompt rein → sofort Transformer → Wort raus → Wort raus → …

Kein Nachdenken. Einfach losfeuern. Schnell, günstig, gut für einfache Anfragen.

🧩 Thinking-Modus

KI macht erst „stille" Transformer-Durchläufe (nicht sichtbar), prüft eigene Antwort, startet ggf. neuen Gedankengang — dann erst spricht sie.

Wie ein Mensch, der kurz die Luft anhält und nachdenkt, bevor er antwortet.

🔧 Wann Reasoning nutzen?
  • Ja: Kostenvoranschläge mit vielen Bedingungen, juristische Analysen, Mathematik, mehrstufige Planung
  • Nein: Einfache E-Mails, kurze Texte, Standard-Übersetzungen — da ist Standard schneller und günstiger
💡 Kernlearning
  • Thinking = Reasoning = Extended Thinking = Effort — alle Begriffe bedeuten dasselbe
  • Bei schlechten Antworten: zuerst Reasoning-Aufwand erhöhen, dann Prompt verbessern
  • Mehr Rechenzeit ≠ immer besser — für einfache Fragen übertrieben

Tools & Harness — der Iron-Man-Anzug

Wie Tools technisch funktionieren:

  1. KI schreibt im Transformer-Lauf: tool_use: research
  2. Der Harness (normales Programm) erkennt das Codewort
  3. Harness führt die Suche aus und gibt das Ergebnis zurück
  4. KI verarbeitet das Ergebnis und antwortet

Tools in modernen Harnessen:

  • 🔍 Internetrecherche
  • 💻 Code-Ausführung / Terminal
  • 📁 Dateien lesen & schreiben
  • 🖥️ Browser Use / Computer Use
  • 📸 Bildgenerierung
💡 Kernlearning
  • KI ohne Werkzeuge = kluge Person ohne Arme
  • Der Harness gibt der KI Hände — erst dann kann sie echte Arbeit erledigen
  • Claude.ai = Fiat Punto | Claude Code = Rennwagen auf der Strecke

Context Window — das Kurzzeitgedächtnis

Was ist das Context Window?

Das „Kurzzeitgedächtnis" der KI für eine Konversation. Alles, was Claude in einer Sitzung weiß — Deine Fragen, seine Antworten, geladene Dateien, Tool-Ergebnisse — liegt darin.

200.000 Token ≈ 150.000 Wörter ≈ ganzer Roman.

Auto-Compacting

Wenn das Context Window fast voll ist, fasst Claude die bisherige Konversation automatisch zusammen und gibt Platz frei. Die Zusammenfassung bleibt, Details gehen verloren.

→ Deshalb ist das KMS so wichtig: wichtiges Wissen muss dauerhaft in Dateien gespeichert werden.

💡 Kernlearning
  • Beim Schließen des Chats: alles weg — Claude erinnert sich an nichts
  • Auto-Compacting ist automatisch, verlustbehaftet und nicht umkehrbar
  • Lösung: Wissen in Dateien persistieren → KMS (Session 3)

Admin-Modus & Terminal-Flags

🔐 Claude ohne Rückfragen starten

Im normalen Modus fragt Claude vor jeder Dateioperation nach Genehmigung. Der Admin-Modus hebt das auf. Empfehlung: erst nutzen, wenn man den normalen Modus kennt.

Die drei wichtigsten Flags:

claude --dangerously-skip-permissions # Admin-Modus — keine Ja/Nein-Fragen mehr # Unten im Terminal: "Bypass Permissions" claude --continue # Letzten Chatverlauf fortsetzen claude --resume # Alle Chatverläufe dieses Ordners anzeigen

Flags lassen sich beliebig kombinieren.

Chathistorie ist ordnergebunden:

# Im Ordner Desktop gestartet: claude --resume → zeigt Chats vom Desktop # Im Ordner Synclaro gestartet: claude --resume → zeigt andere Chats

Die Chathistorie wird in einer Datei im aktuellen Ordner gespeichert.

🔧 Praxisfall — Fabians Bagger-Manual

Fabian musste ~30 Mal auf „Ja" klicken, weil Claude jede Datei-Operation bestätigen wollte. Lösung: Admin-Modus einschalten. Claude arbeitet dann das gesamte 180-Seiten-PDF durch, ohne Unterbrechung. Vorgehen: claude --dangerously-skip-permissions → Prompt mit Aufgabe → durchlaufen lassen → Ergebnis prüfen.

💡 Kernlearning
  • Flag-Name merken: --dangerously-skip-permissions — einmal notieren, dann per ↑ Pfeiltaste abrufbar
  • Admin-Modus erst nutzen, wenn man das Verhalten von Claude kennt und einschätzen kann
  • Chathistorie ist ordnergebunden — --resume zeigt immer nur Chats des aktuellen Ordners

CLAUDE.md — der User-System-Prompt

📄 Was ist CLAUDE.md?

Der User-System-Prompt ist die einzige der drei System-Prompt-Ebenen, die wir selbst editieren können. Er steht noch vor unserem ersten Chat-Prompt und gibt Claude dauerhaftes Wissen über uns. Er wird in einer Datei namens CLAUDE.md gespeichert.

Ladereihenfolge:

  1. Globale CLAUDE.md (immer, egal wo)
  2. CLAUDE.md auf Desktop (wenn Claude dort startet)
  3. CLAUDE.md im KI-Ordner (wenn Claude dort startet)

Dateien unterhalb oder neben dem aktuellen Ordner werden NICHT geladen.

Strategie-Empfehlung (Marco):

  • Einfach: Nur eine globale CLAUDE.md — alles an einem Ort
  • Modular: Globale für immer-gültige Infos + lokale für Themenordner
  • Max. 2 KMS-Systeme pro Computer
"Eigentlich musst du nie mehr selbst in Dateien reinschreiben. Die Zeit ist vorbei. Du sagst Claude, was er da reinschreiben soll."
💡 Kernlearning
  • CLAUDE.md ist der einzige Dateiname, der automatisch geladen wird — alle anderen .md-Dateien müssen explizit geöffnet werden
  • Globale CLAUDE.md = die Mutter aller CLAUDE.mds — wird immer geladen
  • Inhalt: beliebig — von Claude selbst schreiben lassen (z. B. per WhisperFlow einsprechen)

"Wir haben mit Claude einen äußerst fähigen Mitarbeiter, den wir auf dem Bordstein vor unserer Firma getroffen haben. Super smartes Kerlchen. Aber er weiß über unsere Firma nichts." — Das Logbuch macht ihn zum eingearbeiteten Mitarbeiter, der Bescheid weiß.

  • Die globale CLAUDE.md trägt den Verweis auf den KMS-Pfad — jede Claude-Session kennt das KMS
Level 2 — /post-Command: zweite Sicherheitsstufe

Das Problem:

"LLMs haben wie wir Menschen eine Konzentration, die mit der Zeit sinkt. Eine Anweisung von vor 5 Stunden wird nicht mehr so konsequent befolgt. Das unterscheidet eine KI von einem deterministischen Skript."

Die CLAUDE.md-Anweisung allein reicht nicht — bei langen Sessions wird sie „überschrieben" durch neuere Inhalte im Context Window.

Die Lösung: /post

# Am Ende jeder Sitzung eingeben: /post # Claude prüft dann: # - Wurden alle Aktivitäten geloggt? # - Fehlen Einträge? # - Alles vollständig? # - Fehlende Einträge werden nachgeholt.

Der /post-Skill ist ein eingepackter Prompt. Er wird einmal erstellt und dann per /post aufgerufen.

🔧 Das Feierabend-Ritual

Vor dem Schließen von Claude Code immer: /post eingeben → Claude überprüft die Session → fehlende Logs werden nachgetragen → dann schließen. Die Kombination aus CLAUDE.md (passiv) und /post (aktiv) macht das Logging robust.

💡 Kernlearning
  • Skills = wiederverwendbare Prompts, gespeichert in einer Konfigurationsdatei
  • Nach dem Erstellen eines Skills: Claude einmal neu starten (wie ein frisch installiertes Programm)
  • /post ist das Feierabend-Ritual — schützt das KMS vor Lücken
Level 3 — Indexierung: Das Kapitelverzeichnis

Das Problem ohne Index:

Ein Logbuch mit 500 Einträgen vollständig in das Context Window laden = riesige Token-Verschwendung. Bei jedem Aufruf würde die KI Tausende von Tokens lesen, nur um einen Eintrag zu finden.

"Wie ein Roman, der 1.000 Seiten hat. Ohne Inhaltsverzeichnis muss man immer alles lesen." — Marco

Das Such-Protokoll mit Index:

  1. Nur die ersten ~30 Zeilen laden (der Index)
  2. Stichwort suchen → Stempel identifizieren
  3. Volltext-Suche nach z. B. 0236
  4. Findet exakt 2 Treffer: Index-Zeile + Eintrag
  5. Nur diesen Abschnitt laden
# So sieht der Index aus (oben in der Datei): ## INDEX [LOG-0001] — KMS-Grundeinrichtung: Ordner und Logbuch angelegt [LOG-0002] — /post-Command als zweite Sicherheitsstufe eingerichtet [LOG-0003] — Logbuch-Index konzipiert und implementiert ... [LOG-0236] — Bauvorhaben-Management: neues System eingerichtet --- # Suche nach "Bauvorhaben" → findet LOG-0236 # Suche nach "0236" → exakt 2 Treffer → Eintrag laden
💡 Kernlearning
  • Warum Stempel statt Seitenzahlen? In digitalen Dateien können Zeilen verschoben werden — der Stempel bleibt immer eindeutig
  • Jeder Stempel kommt genau zweimal vor: im INDEX und beim Eintragskopf
  • Das ist Context Engineering: Nur das Nötige laden, nicht die ganze Datei
Level 4 — learnings.md & decisions.md
"Learnings und Entscheidungen sind lebensbestimmende Events in unserem Leben. Sie verdienen es, nochmal irgendwo anders separat festgehalten zu werden." — Marco Heer

💡 learnings.md — Format

[L-0001] Datum : 2026-05-13 Kategorie : KI-Verhalten Learning : 2–5 Sätze. Klar verständlich ohne das Logbuch zu lesen. Kontext : Wie/Wo/Warum gelernt? 1–3 Sätze. → Log-Referenz: [LOG-0001] ← PFLICHT

🎯 decisions.md — Format

[D-0001] Datum : 2026-05-13 Bereich : Wissensmanagement Entscheidung : Was wurde beschlossen? 2–5 Sätze. Begründung : Warum? 1–3 Sätze. Auswirkung : Was ändert sich? 1–3 Sätze. → Log-Referenz: [LOG-0001] ← PFLICHT
🔧 Praxisfall — Marco's Zwei-Jahre-Erinnerung

"In zwei Jahren entscheide ich mich, etwas zu machen. Claude sagt mir: 'Laut Decision 534 wolltest du das eigentlich niemals mehr machen. Der Grund steht in Log-Eintrag 302 — wir sind damals auf die Schnauze gefallen. Willst du das wirklich tun?'"

💡 Kernlearning
  • Logbuch = „Warum und Wie" (Vollständigkeit) · Learnings/Decisions = „Was und Ergebnis" (Schnellzugriff)
  • Jeder Eintrag hat eine Pflicht-Quereferenz zurück zum Logbuch — ausnahmslos
  • Beide Dateien haben ebenfalls einen Index am Anfang

KMS als gemeinsames System — Mensch & KI

🤝 Die zwei Grundprinzipien

Prinzip 1: Gemeinsames Eigentum

  • Der Mensch trägt ein: Persönliche Erkenntnisse, Entscheidungen, Erlebnisse
  • Die KI trägt ein: Aktionen, erkannte Fehler, Entscheidungen im Auftrag
  • Fehler festhalten = kein Versagen — es ist der Kern des Lernprozesses
"‚Ich habe einen Fehler gemacht' ist kein Versagen — es ist ein wertvoller Eintrag."

Prinzip 2: Erst nachschlagen, dann handeln

Vor jeder neuen Aufgabe konsultiert Claude das KMS:

  1. decisions.md-Index prüfen — Entscheidungen zu diesem Thema?
  2. learnings.md-Index prüfen — Fehler, die sich nicht wiederholen sollen?
  3. Bei Treffern: Eintrag laden → dann erst anfangen
"Das KMS ist kein Archiv, das verstaubt. Es ist aktives Arbeitswerkzeug."

💡 Das große Bild
  • Studien: Mitarbeiter verbringen 40 Minuten/Tag mit Informationssuche → mit KMS auf ein Zehntel reduzierbar
  • Wissensmanagement → Erfahrungsmanagement: KI greift auf gespeicherte Erfahrungen zurück, bevor sie handelt
  • Heute = Stufe 1 von 5. Nächste Stufen: Vektordatenbanken, Server, weltweit erreichbar
  • "Das ist State of the Art. Das ist, was moderne KI-Nutzung ausmacht." — Marco Heer

🍽️ Die Restaurant-Metapher (Marco Heer)

API = der Kellner. Du (Programm A) sitzt am Tisch. Die Küche (Programm B) kann kochen. Aber du redest nicht direkt mit der Küche — du redest mit dem Kellner. Er nimmt deine Bestellung und bringt das Ergebnis. Du musst nicht wissen, wie die Küche intern funktioniert.

Endpoint = ein Gericht auf der Speisekarte. OpenAI-Beispiele:

  • /chat/completions — GPT antwortet auf Text
  • /audio/speech — Text → MP3 (TTS)
  • /images/generations — Text → Bild
  • /embeddings — Text → Vektoren (für Suche)

API-Key = digitaler Ausweis. OpenAI muss wissen, wer fragt — Abrechnung in Tokens. Geheim halten wie ein Passwort.

Praxisbeispiel DATEV: Buchhaltungssoftware schickt Daten per API an DATEV-Server. Das Ergebnis kommt als JSON zurück. Das Programm muss DATEV nicht "verstehen" — es kennt nur den Endpoint.

Die Architektur — Zwei Ebenen

🐙
GitHub Repository

Die einzige Source of Truth für alles geschriebene Wissen. Kein Device hält lokale Wahrheit. Alle lesen und schreiben via Git. Jeder Push auf main triggert automatisch das Re-Embedding der Vektordatenbank.

🔍
Supabase pgvector

Die vektorisierte Spiegelung für semantisches Retrieval. Jedes Dokument wird in 1024-dimensionale Vektoren übersetzt. Suche nach Bedeutung statt nach Stichworten. Embedding-Modell: Voyage-4-large.

Kernprinzip 1 — INVASIV vs. ADDITIV

🔄 Jede Datei im System ist eines von beidem — das bestimmt, wie sie geschrieben wird
🔴 INVASIV = Snapshot

Wird bei jeder Aktualisierung komplett überschrieben. Keine Historie innerhalb der Datei — nur der aktuelle Stand zählt.

  • status-quo.md — Nur Gegenwart, max 100 Zeilen
  • wissen/README.md — Routing-Tabelle, immer aktuell
🟢 ADDITIV = Gedächtnis

Nur anfügen, nie ändern. Bestehende Einträge werden nicht verändert oder gelöscht. Das ist das unveränderliche Archiv.

  • logbuch.md — Jeder Eintrag bleibt für immer
  • DECISIONS.md — Kein Zeilenlimit, kumulativ
  • LEARNINGS.md — CRITICAL-Einträge werden NIE archiviert

Kernprinzip 2 — PERMANENT vs. VOLATIL

🗿 PERMANENT

Kernwissen, das sich selten ändert: Strategie, Methodik, Frameworks, Unternehmensstruktur. Wird nur ergänzt, nie ersetzt. Wenn sich Kernwissen fundamental ändert → als Entscheidung in DECISIONS.md dokumentieren.

⚡ VOLATIL

Wissen, das veraltet: Metriken, Kampagnen-Daten, Preise, Marktanalysen. Darf überschrieben werden. Staleness-Warnung wenn Letzte Aktualisierung > 30 Tage alt. Änderungen werden als Learning dokumentiert.

Schritt 1 — Global oder Lokal?

🌐 GLOBAL (empfohlen)

Die Verhaltensregeln werden in ~/.claude/CLAUDE.md geschrieben. Egal in welchem Ordner Claude gestartet wird — er pflegt das KMS immer. Alle Dateien in einem zentralen Ordner.

💡 Empfehlung

Für die meisten Nutzer die richtige Wahl. Das KMS ist projektübergreifend und immer aktiv.

📁 LOKAL

Verhaltensregeln nur in der CLAUDE.md des jeweiligen Projektordners. Gilt NUR wenn Claude in diesem Ordner gestartet wird.

Sinnvoll wenn: Verschiedene Projekte unterschiedliche KMS-Strukturen benötigen.

Das Werkstatt-Prinzip — KRITISCH

🔨 Permanent pflegen — NICHT erst am Ende

Das Wissensmanagement folgt dem Werkstatt-Prinzip: Die Werkstatt wird WÄHREND der Arbeit permanent sauber gehalten, nicht erst nach Feierabend aufgeräumt.

❌ Falsch
  • 4-5 Sessions durcharbeiten
  • Dann erst alles dokumentieren
  • Ergebnis: Erste Sessions fast vergessen
  • /post als "großes Aufräumen" missbrauchen
✅ Richtig
  • Jede wesentliche Aktion SOFORT in log.md
  • Jede Entscheidung SOFORT in decisions.md
  • Jedes Learning SOFORT in learnings.md
  • /post ist nur der letzte Kontrollgang

Slash-Commands — /pre und /post

🚀 /pre [optionales Thema]

Intelligente Session-Initialisierung. Lädt relevanten Kontext, ohne das Kontextfenster zu vermüllen.

  1. git pull — neuesten Stand holen
  2. status-quo.md laden, Staleness prüfen
  3. Thematische Filterung: Mit Thema → nur relevante Bücher. Ohne Thema → genereller Überblick
  4. Strukturierter Überblick ausgeben
/pre ← Genereller Überblick /pre SEO-Strategie ← Nur SEO-relevantes /pre Kalkulation Holzbau ← Spezifisch
🏁 /post

Letzter Kontrollgang am Ende einer Session. Prüft Vollständigkeit, füllt Lücken, synchronisiert mit GitHub.

  1. Vollständigkeits-Prüfung: Alle Aktionen geloggt? Format korrekt? Index aktuell?
  2. Lücken füllen — falls etwas fehlt: jetzt nachtragen
  3. Ergebnis-Bericht — was fehlte, was ergänzt wurde
  4. git commit + push — alles zu GitHub synchronisieren
⚠️ Wichtig

/post ist KEIN Ersatz für permanentes Pflegen. Es ist die Qualitätssicherung, nicht der Hauptprozess.

GitHub-Synchronisation — Multi-Device

🔄 Der Sync-Ablauf
/pre auslösen → git pull (neuester Stand von GitHub) → Kontext laden → Arbeiten... Werkstatt-Prinzip: permanent pflegen /post auslösen → Vollständigkeits-Check → git add + commit + push (Änderungen zu GitHub)
💡 Goldene Regeln für Multi-Device
  • Immer /pre am Anfang und /post am Ende einer Session
  • ADDITIV-Dateien (log, decisions, learnings) können grundsätzlich konfliktfrei zusammengeführt werden
  • INVASIV-Datei status-quo.md kann Konflikte erzeugen — im Zweifel die neuere Version behalten
  • Auf weiteren Computern: Repository klonen → CLAUDE.md erstellen → /pre und /post kopieren → fertig

1. Hauptthema — Warum "Prüf nochmal alles" nicht funktioniert

🕵️ Das eigentliche Problem ist kein Qualitäts-, sondern ein Awareness-Problem

KI findet und behebt Fehler dort zuverlässig, wo sie hinschaut. Das Problem sind Stellen, an die sie schlicht nicht hinschaut — der Prompt "Schau nochmal, ob alles passt" sagt ihr nicht, WOHIN sie schauen soll.

🚫 Dieser Prompt ist verboten: „Jetzt schau noch mal, ob alles in Ordnung ist und gut läuft."
✅ Der empfohlene 6-Schritte-Workflow
SchrittAktion
1NICHT fragen: "Schau nochmal, ob alles passt" (siehe oben)
2/post — alles Gebaute/Gelernte ins Wissensmanagement sichern
3/compact — Kontextfenster verdichten, Claude "frisch im Kopf" für die Testphase
4Codebase-Analyse — jede Funktionalität, jeden Datenfluss sichtbar machen
5Autonome Testung per Browser-Usage inkl. MCP-Datenbank-Verifikation

Schritt 4 — Codebase-Analyse (reine Awareness-Herstellung):

Führe jetzt nochmal eine Codebase-Analyse durch, in der du dir nochmal
jede Funktionalität auf den Schirm rufst, die es gibt. Dazu gehört:
jeder Datenspeicherort
jeder Datenfluss
jede Datenverarbeitung
jede User-Interaktion im Frontend (also alle Knöpfe, alle Eingabefenster,
alle Links, alles, was irgendwie testbar ist)

Schritt 5 — Autonome Testung mittels Browser-Usage (der zentrale Prompt der Session):

Autonome Testung mittels Browser-Usage von Frontend: Sämtliche
Funktionalitäten, die geklickt werden können oder getippt werden können,
müssen entweder über Chrome oder über Playwright komplett autonom
getestet werden. Außerdem müssen dann alle Konsequenzen gemessen werden:
Datenbank-Einträge, Weiterleitungen, Erwartungshaltungen, die man eben
einfach haben darf, wenn man im Frontend irgendetwas macht. Außerdem
müssen alle weiteren Features, die dieses Programm bzw. diese Anwendung,
die wir gemacht haben, hat, getestet werden. Erstelle ein umfangreiches
Testing-Szenario, wie es von einem professionellen Softwarehaus
erschaffen würde, um sowohl Datensicherheit als auch User Experience und
Datenflüsse komplett autonom durchzutesten. Teste das komplett autonom
durch, gib das Testing-Protokoll dann noch mal an einen Subagenten, um
zu überprüfen, ob wirklich alles gut gelaufen ist.

Tool-Use-Muster: Browser Usage → MCP Supabase-Check → ggf. Fehlerbehebung. Claude öffnet dabei selbst einen Browser, klickt jeden möglichen Klick durch, prüft per MCP direkt in Supabase, ob z. B. ein neu angelegter Datensatz wirklich angekommen ist, und liest zusätzlich die Entwicklerkonsole auf Fehlermeldungen aus.

"Diese ganze Testung, die ihr sonst eigentlich mit Klicken und Machen tun müsstet, kann man von KI abverlangen. Dauert halt ein Weilchen länger, dann geht ihr halt einen Kaffee trinken." — Marco Heer

Ersetzt laut Marco Heer ca. 95 % des eigenen manuellen Testings. Wichtig: die Prompt-Kette nicht blind wiederverwenden, sondern selbst nachformulieren — das trainiert den Transfer auf eigene Projekte.

5. Datenschutz-Exkurs — Der Entscheidungsbaum (PII, AVV, DPA)

PII (Personally Identifiable Information) = alles, was Rückschluss auf eine konkrete Person zulässt. Zwei Vertragswerke regeln den Umgang damit:

KürzelBedeutungRegelt
AVVAuftragsverarbeitungsvertragSpeicherung von Daten
DPAData Privacy AgreementVerarbeitung von Daten

Wichtig: Sobald Daten in den USA gespeichert/verarbeitet werden, wird die Wirksamkeit eines AVV nach aktueller DSGVO-Linie faktisch ausgehebelt.

🌳 Der Entscheidungsbaum
1. Dürfen wir die Daten so verarbeiten/speichern?
Ja → weitermachen · Nein → weiter zu 2.

2. Können wir AVV/DPA abschließen, um es zu dürfen?
Ja → weitermachen (Ausnahme z.B. Gesundheitsdaten) · Nein → weiter zu 3.

3. Verarbeitung/Speicherung auf konforme Lösung umstellen:
→ Lokal hosten (lokale Supabase-Instanz, lokales KI-Modell/OCR)
→ Datensatz vorher anonymisieren/pseudonymisieren
→ Proxy-Modell: lokales Modell prüft Prompt vorab, nur Unkritisches
geht an die öffentliche/US-KI
✅ Praktisch unkritisch

Reines Programmieren/Testen, private Nutzung mit eigenen Testdaten — hier spielt DSGVO praktisch keine Rolle.

⚠️ Relevant wird es bei…

Echten (Kunden-)Daten im Firmeneinsatz — und deutlich kritischer beim Verkauf an Dritte.

Mistral ist EU-ansässig — sobald ein AVV mit Mistral besteht, ist die OCR-Verarbeitung weitgehend datenschutzkonform nutzbar.

🔑 API-Keys niemals im HTML/Frontend: Wird das HTML weitergegeben, "fliegt der Schlüssel mit durch die Gegend". Lösung: API-Calls über eine Edge Function (Backend) statt direkt aus dem Frontend.

7. Vollständige DMS-Architektur (am Whiteboard erarbeitet)

🏗️ Die Bausteine und ihre Reihenfolge
1. PROGRAMMIERER (Claude Code) → baut Frontend + richtet Datenbank ein

2. FRONTEND (HTML, z.B. auf Netlify) → zeigt Daten aus Supabase,
Drag & Drop-Fenster für Dokumente

3. Datei landet NICHT direkt in der Datenbank, sondern im
DATEISPEICHER (Supabase Storage) → triggert eine EDGE FUNCTION

4. EDGE FUNCTION:
Schritt 1 — OCR: Mistral OCR (OCR Latest, NICHT Pixtral!) liest Text
Schritt 2 — DSGVO-Prüfung: sensible Daten? ja/nein
Schritt 3 — Kategorisierung per KI-Aufruf (Rechnung/Angebot/...)
Schritt 4 — Handlungsbedarf: z.B. Rechnung > 50€ → Telegram-Freigabe

5. Edge Function schreibt Ergebnis zurück in die DATENBANK

6. CLAUDE ↔ SUPABASE über MCP-Server: Direktzugriff ohne Kopieren

7. Optional: Edge Function löst Telegram-Nachricht aus
(z.B. "Monatliche Auswertung an Marvin senden")

Eine Edge Function ist kein fertiges Anbieter-Produkt, sondern selbst geschriebener Code (wie ein Python-/JS-Skript), der bei Supabase läuft — dort, wo die Daten sind, unabhängig von der eigenen Hardware, abgesichert durch den mit Supabase geschlossenen AVV.

"Datenspeicherung hat in einer Datenbank zu erfolgen, Dateispeicherung hat in einem Dateispeicher zu erfolgen. KI-Calls haben nicht aus dem HTML raus zu erfolgen, sondern aus einer Edge Function heraus. Das ist der Unterschied zwischen Vibe-Coding und professionalisierter Architektur." — Marco Heer

🛡️ Sicherheitsprüfung — Security Awareness

Vollständiges Security-Audit Vor dem Livegang
Prüft die häufigsten Schwachstellen von KI-gebauten HTML/JS-Web-Apps: offengelegte API-Keys, ungeschützte Datenbanken, fehlende Authentifizierung.
Starte einen Subagenten für ein gründliches Sicherheits-Audit dieser Anwendung. Gehe vor wie ein Senior-Security-Experte für Web-Apps und prüfe systematisch auf einwandfreie Sicherheitsvorkehrungen: 1. GEHEIMNISSE / API-KEYS - Liegt irgendwo ein API-Key, Passwort oder Token im Frontend-Code (HTML/JS), das mit ausgeliefert wird? Solche Schlüssel dürfen niemals im Client landen. - Werden KI-/API-Calls direkt aus dem HTML gemacht? Falls ja: auf eine Edge Function oder serverseitige Function umstellen, wo der Schlüssel verschlüsselt/als Umgebungsvariable liegt. 2. DATENBANK-ZUGRIFF (falls Supabase o.ä.) - Ist Row Level Security (RLS) aktiv? Gibt es versehentlich offene Policies (z.B. "anon" darf alles lesen/schreiben/löschen)? - Kann ein Fremder mit der reinen Frontend-URL über den öffentlichen Key die komplette Datenbank auslesen oder verändern? 3. AUTHENTIFIZIERUNG - Ist ein echter Login vorhanden oder nur ein Klartext-Passwortvergleich im Code? - Werden Sitzungen/Token sicher gehandhabt? 4. DATENFLÜSSE & EINGABEN - Werden Nutzereingaben validiert? Bestehen Injection-Risiken? - Werden Fehler beim Speichern still verschluckt (scheinbar erfolgreich, obwohl nichts gespeichert wurde)? 5. HOSTING / DEPLOYMENT - Liegen Secrets korrekt als Umgebungsvariablen (Netlify/Supabase/GitHub) und nicht im Repository? Erstelle einen priorisierten Befund (kritisch / hoch / mittel / niedrig) mit konkreter Fundstelle und jeweils einem umsetzbaren Lösungsvorschlag. Behebe im Anschluss nach meiner Freigabe die kritischen Punkte, ohne bestehende Funktionalität oder Optik zu verändern.
💡 Merke (Session 7): „Ein HTML mit API-Schlüssel einfach rumreichen ist nicht gut." Sobald ein Frontend eine KI/API direkt anspricht, muss der Schlüssel über eine Edge Function laufen — nie im Client.

🔒 DSGVO-Prüfung — Datenschutz

DSGVO-Audit per Subagent Am Projektende
Prüft die fertige Anwendung gegen die DSGVO — anhand des Entscheidungsbaums PII → AVV/DPA → konforme Lösung aus Session 7.
Starte einen Subagenten, der alles, was wir in dieser Anwendung gebaut haben, DSGVO-kritisch beleuchtet und mir sofort Bescheid gibt, wie wir es umsetzen müssen — damit es nicht nur technisch gut gedacht ist, sondern auch datenschutztechnisch. Prüfe entlang dieses Entscheidungsbaums: 1. Werden personenbezogene Daten (PII) verarbeitet oder gespeichert? PII = alles, was einen Rückschluss auf eine konkrete Person zulässt (Name, Geburtsdatum, Ausweis-/Kundennummer in Verbindung mit Namen usw.). Liste konkret auf, welche Datenfelder betroffen sind. 2. Dürfen wir diese Daten so verarbeiten/speichern? - Falls nicht: Lässt es sich durch einen Auftragsverarbeitungsvertrag (AVV, regelt Speicherung) bzw. ein Data Privacy Agreement (DPA, regelt Verarbeitung) mit den genutzten Anbietern erlauben? Nenne die betroffenen Dienste (z.B. Supabase, Anbieter der KI/OCR, Hosting) und ob dort ein AVV/DPA verfügbar ist. - Achtung: Werden Daten in den USA gespeichert/verarbeitet, wird ein AVV nach aktueller DSGVO-Linie faktisch ausgehebelt. Weise mich darauf hin, wo US-Dienste im Spiel sind. 3. Falls weiterhin nicht zulässig: Schlage konkrete konforme Alternativen vor — - lokal hosten (z.B. lokale Datenbank-Instanz, lokal laufendes KI-/OCR-Modell), - Datensatz vorher anonymisieren/pseudonymisieren (identifizierende Merkmale entfernen), - Proxy-Modell (lokales Modell prüft jeden Prompt vorab; nur unkritische Inhalte gehen an eine öffentliche/US-KI). Bitte beziehe dich auf die relevanten DSGVO-Artikel (u.a. Art. 5, 25, 28, 30, 32, 44) und gib mir am Ende eine priorisierte Maßnahmenliste. Du bist keine Rechtsberatung — es geht um eine solide technische Datenschutz-Einschätzung.
